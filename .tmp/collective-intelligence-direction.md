# 集合知化の方向性 — チーム合意用ブレスト結果

ブレインストーミングで決まった「チームの向かうべき方向性」を、漏れなくまとめたもの。
この文書をベースに、チームに共有し、上司への依頼を設計する。

> **📝 更新履歴**
> - 初版: チームブレスト結果を整理
> - **改訂版: 4視点レビュー（現場・上司・技術・批判派）の結果を反映** ← **重要な修正あり。セクション9を必読**

## 1. 議論の前提（決まったこと）

### 1.1 この提案の「相手」
- **第一の相手**: 自分のチーム（5〜10人程度）に、明日から一緒にやる提案
- **第二の相手**: 方向性が決まったあと、上司に「実現にはこれが必要」と依頼
- したがって2段階: **チーム合意 → 上司への必要物の依頼**

### 1.2 チームの温度感
- **全員前向き**。すでに各自でバイブコーディングしている。あとは方向性を揃えるだけ
- したがって「攻めた方向性」を取れる。スキーム変更で抵抗はない前提

### 1.3 最高の方向性（旗）
- **集合知化**: 個人の知恵がチームの知恵になる仕組みを作る
- 元々の選択肢のうち「ノウハウ資産化／品質の均一化／チームで一つの脳」の3つは同じ軸の別側面と整理
  - ノウハウ資産化 = 知恵を「残す」（時間軸）
  - 品質の均一化 = 知恵を「揃える」（横軸）
  - チームで一つの脳 = 知恵を「掛け合わせる」（化学反応）
- これに対する対抗軸は「スピード」「解放（雑務からの自由）」。今回は集合知化を第一の旗に選んだ

### 1.4 集合知化の「壁」
チームに3つとも欠けている：
1. **場所がない** — 共有したくても「どこに書けばいい？」が決まっていない
2. **習慣がない** — 場所はあっても、誰も書かない。頭の中で完結
3. **時間がない** — 書きたくても余裕がない

### 1.5 現状のインフラ
- **Gitea はすでにある** — サーバー・アカウントも完備
- しかし **「完成コードの保存場所」としてしか使っていない**
- 開発中のコードは **各個人のPCにある**
- Issue も PR もレビューも習慣化していない

### 1.6 GitHub ではなく Gitea を使う理由
1. **データを社外に出せない**（セキュリティ・ポリシー）
2. **コストが厳しい**（GitHub Team の $4/user/月でも予算確保が難しい）
- 上記2つの両方が該当するため、Gitea 継続運用が確定

## 2. 方向性A — Gitea 中心型

### 2.1 一行で言うと
> **「Gitea を『完成コードの墓場』から『チームの脳と作業場所』に変える」**

### 2.2 なぜこの方向性か
- 集合知化の3つの壁（場所・習慣・時間）を同時に解決できる唯一の現実解
- 「共有」を別の作業にしない。**本業の副産物として知識が溜まる構造**を作る
- 追加の大きな投資はほぼゼロ。**既にある Gitea の使い方を変えるだけ**
- 上司への説明もしやすい（「GitHub/Gitea を使います」は誰でもわかる）

### 2.3 検討した他の選択肢
| 方向性 | 内容 | 採用しない理由 |
|---|---|---|
| B. CLAUDE.md 育成型 | 週1の振り返りで CLAUDE.md を更新するだけ | シンプルすぎて「別作業」感が残る。習慣化しない可能性 |
| C. 仕掛け自動化型 | Hooks・Subagents・MCP を駆使した自動化 | 土台がないと動かない。Aの後に乗せるべきもの |

### 2.4 方向性A の5つの要素
1. **日常の開発の「流れ」を Gitea に通す**
   - コードは今まで通り VS Code + Claude Code（手元）で書く
   - 書く前に Issue、書いたあとに PR。**書く作業の入口と出口が Gitea**
2. **Issue に「何を・なぜ」を書く**
   - 壁打ちは手元の Claude Code で行い、**結論を Issue にポストする**
3. **PR レビューを習慣化**
   - 秘書（Claude Code）によるセルフレビュー → 人間レビューの2段階
4. **CLAUDE.md をリポジトリに置いて、チームで育てる**
   - Git でバージョン管理、PR 経由で更新
5. **秘書はローカルで動かす**
   - Gitea 連携は手動コピペ or 簡単な slash command で

## 3. 技術的な骨格

### 3.1 2階層の CLAUDE.md 構造

```
┌──────────────────────────────────────────┐
│ 共通 CLAUDE.md（team-standards リポジトリ）  │
│ ・デザインシステム（Serendie使う）             │
│ ・命名規約・any禁止                           │
│ ・壁打ち→設計→実装の流れ                    │
│ ・セキュリティポリシー                        │
│ ・共通の失敗あるある                          │
└──────────────────────────────────────────┘
                ↑ @import
┌──────────────────────────────────────────┐
│ プロジェクト CLAUDE.md（各リポジトリ）          │
│ ・プロジェクトの目的                          │
│ ・技術スタック                                │
│ ・ディレクトリ構造                            │
│ ・プロジェクト特有の注意点                    │
└──────────────────────────────────────────┘
```

### 3.2 team-standards リポジトリの中身

```
team-standards/
├── CLAUDE.md           ← 常に読まれる基本ルール
├── skills/             ← 呼び出し式の手順
│   ├── create-api-endpoint.md
│   ├── write-test.md
│   └── deploy-checklist.md
├── agents/             ← 専門秘書
│   ├── security-reviewer.md
│   ├── architecture-reviewer.md
│   └── test-coverage-checker.md
├── hooks/              ← 自動化スクリプト本体
│   ├── pre-commit-lint.sh
│   └── post-session-summary.sh
└── mcp/                ← 外部ツール接続設定
    ├── gitea-mcp.json
    └── slack-mcp.json
```

各開発者は `~/dev/team-standards` に clone、週に1回 `git pull`。

### 3.3 各機能の役割分担

| 機能 | 役割 | 置き場所 | いつ読まれるか |
|---|---|---|---|
| **CLAUDE.md** | 基本ルール・背景知識 | team-standards + 各リポジトリ | **常に**自動で |
| **Skills** | 手順・ワークフロー | team-standards + 個別 | **呼び出されたときだけ** |
| **Subagents** | 専門秘書（特殊な役割） | team-standards | 指名されたときだけ |
| **Hooks** | 自動化スクリプト | team-standards（本体）+ 個人設定で有効化 | 特定のイベント時 |
| **MCP** | 外部ツール連携 | team-standards（設定）+ 個人でインストール | ツール使用時 |

### 3.4 CLAUDE.md と Skills の使い分け

| | CLAUDE.md | Skills |
|---|---|---|
| いつ読まれる | 常に | 呼ばれたときだけ |
| サイズ | 短く（200行以内目安） | 長くてもOK |
| 中身 | 常識・ルール・背景 | 特定の手順・チェックリスト |
| 例 | 「any 禁止」「Serendie 使う」 | 「新しい API エンドポイントを作る手順」 |

**原則**: 「常識」は CLAUDE.md、「手順」は Skills。これで CLAUDE.md が肥大化しない。

### 3.5 Skills が集合知化で果たす役割
- 個人が見つけた「良いやり方」を Skill に落とす
- 強制しないが、必要なときに全員が使える
- 例: `/implement-feature` が「壁打ち→Issue→Draft PR→テスト→実装→レビュー」を促す
- **2回以上やる作業は Skill にする**のが目安

## 4. Issue の粒度

### 4.1 Issue を立てる基準
> **「後から『なぜそうなった？』と聞かれたときに、答えが残っていてほしい単位」**

### 4.2 立てる／立てない例

| 種類 | 立てる？ |
|---|---|
| 新しいアプリ・ツール | 必ず立てる |
| 機能追加 | 必ず立てる |
| 既存機能の改善 | 立てる |
| バグ修正（影響大） | 立てる |
| バグ修正（1行・タイポ） | 立てない（直接PR） |
| 試作・実験 | 立てる（結果を残したい） |
| 調査・検討 | 立てる（結論が資産） |
| 決めたいこと（命名規則など） | 立てる（議論の場として） |

### 4.3 親Issue / 子Issue の階層
```
Issue: 新しいアプリ「日報自動化ツール」を作る
 ├─ Issue: [機能] 日報テンプレートを用意する
 ├─ Issue: [機能] Slack から日報を書ける
 ├─ Issue: [調査] どの LLM が安い・速いか比較
 └─ Issue: [決定] データの保存場所をどうするか
```

### 4.4 立ち上げ期のシンプルなルール
> **「30分以上考える作業をやる前に、Issue を立てる。中身は3行でもいい」**

3行テンプレ:
```
何: 〇〇をやりたい
なぜ: 〇〇だから
どうやって: たぶん〇〇な感じ（これから詰める）
```

## 5. 日々の開発フロー

### 5.1 1日の流れ（典型例）

**朝（Gitea）**
1. 自分アサインの Issue を見る
2. 今日やる Issue を1つ選ぶ → コメントで「今日やります」

**作業開始（VS Code + Claude Code）**
3. `git checkout -b feature/issue-42` でブランチを切る
4. Claude Code 起動 → 壁打ち開始
   - Claude は CLAUDE.md と team-standards を自動で読み込み済み
5. 決まった方針を **壁打ちログとして Issue にポスト**（連携ポイント①）
6. コードを書かせる、手直し、テスト

**途中（Gitea）**
7. 最初のコミットができたら push → **Draft PR を作成**
8. 必要に応じて同僚に早めの意見を聞く

**実装を進める（VS Code + Claude Code）**
9. 壁打ちしながら実装を進める
10. セルフレビュー（連携ポイント②）

**レビュー（Gitea）**
11. Draft → Ready for review
12. 人間レビュー → 指摘対応 → 再push

**マージ後**
13. 学びがあれば **CLAUDE.md 更新 PR**（連携ポイント③）
14. Issue クローズ

### 5.2 3つの連携ポイントの実装

#### 連携①: 壁打ちログを Issue に残す
- **最初は手動コピペで OK**（要約を作ってもらって貼るだけ）
- **後に**: スラッシュコマンド `/post-to-gitea-issue` を作る
  - Gitea API は GitHub 互換なので、数十行のスクリプトで実装可能
  - 終了時に「今の議論を Issue #42 にコメント投稿して」と指示すれば自動投稿

#### 連携②: セルフレビュー
- スラッシュコマンド `/review-my-changes`
- `git diff` を取って、Claude Code に「CLAUDE.md 準拠か、テストあるか、過去の失敗事例に該当しないか」を点検させる
- 自動ではないが、「push 前に一回叩く」を習慣にする

#### 連携③: CLAUDE.md 更新
- 小さな PR でチームに提案
- 「こういう時はこうしたほうが良さそう」と3行追記の PR
- レビュー → マージ → 次から全員の秘書が知ってる状態

### 5.3 team-standards の参照方法
各開発者が `~/dev/team-standards` にclone、プロジェクトの CLAUDE.md で:
```markdown
@~/dev/team-standards/CLAUDE.md
@~/dev/team-standards/skills/
```
週1回 `git pull`（自動同期より、能動的な pull が安全）。

## 6. 議論中に整理した重要な論点

### 6.1 「Gitea で開発する」の意味
- ❌ 誤: Web ブラウザ上でコードを書く（非効率）
- ✅ 正: **開発の「流れ」が Gitea を通る**。コードは VS Code + Claude Code（手元）
- VS Code + Claude Code = 職人の作業台
- Gitea = 工場の共同作業場（設計指示・材料置き場・検品・出荷記録）

### 6.2 壁打ちログは自動で残らない
- Claude Code の会話はデフォルトでは手元にしか残らない
- 残すには意図的な仕組みが必要（コピペ or slash command）
- 最初は手動でOK、後から自動化する

### 6.3 Issue の粒度で迷ったら
- 「機能」「アプリ」「改善」「調査」「決定」何でも立てていい
- 基準は **「考えた跡を残したいか」**
- 30分以上考えるなら立てる

### 6.4 CLAUDE.md のバラバラ問題
- リポジトリごとに独立させると、共通ルール（デザイン等）がバラバラになる
- → **team-standards リポジトリで共通化、`@import` で読み込む**
- 共通 vs プロジェクトの判断基準: 「別のプロジェクトでも同じことを言うか？」
- 迷ったらプロジェクト側、後から共通に「昇格」させる

### 6.5 2人以上の並列開発
- **全く問題なし**。Git の基本そのもの
- 各自が独立したブランチで作業
- Claude Code のセッションも別々に動く（干渉しない）
- 1人が複数Issue を並行するのも可能（Boris Cherny は5並列）

### 6.6 マージ衝突の扱い
- **テキストレベルの衝突の8割は Claude Code で解決できる**
- できないこと:
  - 本質的に矛盾する設計判断（人間の判断が必要）
  - セマンティック衝突（Git が検知できない型不整合など）
  - CLAUDE.md 自体の衝突（チームで話し合う必要）
- コツ:
  - こまめに pull
  - PR は小さく・早く
  - 共通ファイルを触るときは声かけ

### 6.7 Hooks と MCP の扱い
- **スクリプト本体は team-standards に置く**（共通化）
- **有効化する設定は個人の `.claude/settings.json`**（各自の好みを許容）
- 「全員強制」したいものと「個人の好み」を分ける

## 7. 上司への依頼リスト

「方向性が決まった。実現には以下が必要」という形で話す材料。

### 7.1 【承認】Claude Code を公式な開発ツールとして使うこと
- 情報セキュリティ的に「社内コードを LLM に送る」ことの承認が必要
- Anthropic のデータポリシー（学習に使わない等）を説明できる材料を準備
- **代替案**: データを外に出せないなら、Azure OpenAI 経由や、オンプレ LLM も検討（この場合設計が大きく変わる）

### 7.2 【予算】Claude Code サブスクリプション費用
- 各メンバーに Claude Pro または Max プラン、あるいは API クレジット
- 概算: 5-10人 × 月額数千〜2万円/人 = 月額数万円〜20万円
- **代替案**: 最初は1-2人の試験運用から、成果を見て全員展開

### 7.3 【時間】移行期間の業務時間確保
- 最初の1〜2ヶ月は新ワークフローに慣れる期間
- 生産性が一時的に落ちる可能性を許容してもらう
- 「学習コスト込みで承認」が明示されていると現場が動きやすい

### 7.4 【承認】新しい開発プロセスを公式化すること
- 「Gitea で Issue 立てる、PR でレビューする」を課の正式ルールに
- 個人の自由な試みではなく、チームとしての標準として扱う

### 7.5 【時間】振り返り時間の業務化
- 週15分の振り返り（CLAUDE.md の更新候補の議論）を業務時間として認める
- 「コードを書いている時間だけが仕事じゃない」ことへの理解

### 7.6 【人】推進役の指名
- 誰か1人（または2人）が旗振り役になる
- 兼務でOK、でも公式に任命されていると動きやすい

### 7.7 【場合により】Gitea サーバーの増強
- 既存 Gitea で問題ないはずだが、容量・負荷を念のため確認
- 不足していれば増強の承認

## 9. 4視点レビューの結果と方向性の修正

ブレスト結果を4エージェント（現場・上司・技術・批判派）に並列で叩いてもらった結果、**方向性Aは維持しつつ、重要な修正が必要**となった。以下、修正内容をまとめる。

### 9.1 4視点共通で指摘された致命傷

> **「推進役1人の献身で回っている状態を、誰も失敗と認識できないまま半年経過するシナリオ」**

- **批判派**: 属人的献身を成果と誤認する罠
- **現場**: 推進役の休暇中にチーム全体が止まる
- **上司**: リーダー依存リスク
- **技術**: バージョニング・運用が人依存になる

→ **設計全体を「1人が抜けても回る」前提で作り直す必要がある**

### 9.2 修正1: スコープの縮小（最重要）

| 項目 | 旧 | 新 |
|---|---|---|
| 対象 | 5〜10人の課全員で一斉導入 | **3名のパイロット、3ヶ月運用** |
| 展開 | 最初から全員 | 成功したら残り7人に展開 |

**理由**: 上司目線では「組織改革案件」に分類され、課長の一存で決められない。批判派も「全員で始めるのは爆弾」と指摘。3名なら失敗コストが 1/3。

### 9.3 修正2: 推進役依存の罠への対策（最重要）

以下を**最初から**設計に組み込む：

1. **輪番制の徹底**
   - CLAUDE.md 更新当番を週替わり
   - 振り返り議事のまとめ役も週替わり
   - team-standards レビュアーを **CODEOWNERS 2人体制**（1人だと休暇で止まる）

2. **属人化指標を KPI に**
   - 「推進役以外の人が触った PR の比率」
   - 「推進役以外の人による CLAUDE.md 更新の数」
   - これが低いままなら「集合知化」は名ばかり

3. **2週間休暇テスト**
   - 3ヶ月目に、推進役を意図的に 2週間休ませる
   - 回らなければ設計をやり直す（文化ではなく一人の献身だったと判定）

### 9.4 修正3: 技術設計の修正

4視点の「技術実装」エージェントが公式ドキュメントを確認し、構想の一部に致命的誤りを発見：

#### 致命的誤り
- **「team-standards を直接参照」は Skills/Subagents では動かない**
  - Skills は `~/.claude/skills/` か `.claude/skills/` か Plugin のみ読み込み対象
  - Subagents も同様
  - `--add-dir` で追加したディレクトリからは読み込まれない

#### 正解
- **最終形は Claude Code Plugin 化**
  - Anthropic公式が skills/agents/hooks/commands/MCP を一括配布する仕組み
  - バージョニング・破壊的変更・ロールバック問題も同時解決

#### 段階的な構築ステップ（推奨）
| Phase | 期間 | 内容 |
|---|---|---|
| Phase 1 | 1日 | team-standards リポジトリ + CLAUDE.md + `@import` で最小開始 |
| Phase 2 | 1日 | Gitea 公式MCP 導入、壁打ちログ slash command |
| Phase 3 | 3-5日 | `.claude/rules/` + Hooks（symlink公式サポート活用） |
| Phase 4 | 1-2週 | **Plugin 化**（最終形） |
| Phase 5 | 必要時 | セルフレビュー（まず公式 `/simplify` から） |

#### その他の技術的補正
- **Gitea 公式 MCP が存在する**: [gitea/gitea-mcp](https://gitea.com/gitea/gitea-mcp) → claude-code-action 不要で実現可能
- **`@~/dev/team-standards/CLAUDE.md` の絶対パス import は公式サポート済み**（ただし初回の承認ダイアログを拒否したら二度と復活しない）
- **スクリプトは `.sh` ではなく Node.js か Python で書く**（Windows/Unix差異を避ける）
- **Gitea トークンは `${GITEA_TOKEN}` env var 必須**、team-standards に直書き禁止
- **`.claude/rules/` の活用**（symlink 公式サポート、元の構想から漏れていた仕組み）

### 9.5 修正4: 運用ルールの明文化（形骸化防止）

現場目線エージェントが「曖昧なルールは3週間で崩れる」と指摘。以下を最初から明文化する：

#### Issue 起票ルール（時間ベース）
~~「考えた跡を残したい単位」~~（主観的すぎる）
→ **「15分以上かかる作業は Issue 必須、それ未満は PR のみで OK」**

#### 壁打ちログの記録形式
~~ログ全体をコピペ~~（3日で誰もやらない）
→ **4項目テンプレで要約を Issue にポスト**

- 案B: ...

#### CLAUDE.md 更新の仕組み

### 9.6 修正5: 撤退条件の事前明記

上司目線と批判派の両方が「撤退条件がないのが一番怖い」と指摘。以下を事前に決めておく：

#### 3ヶ月時点のチェック
- 推進役以外の人の CLAUDE.md 更新 PR が **5件以上**あるか
- 壁打ちログの 4項目テンプレが **週平均2件以上**蓄積されているか
- team-standards の CODEOWNERS 2人とも機能しているか
- 参加者 3名がワークフローに「慣れた」と感じているか

#### 6ヶ月時点のチェック
- 2週間の推進役休暇テストを実施し、ワークフローが回るか
- 集合知化の実感（アンケート or インタビュー）
- Before/After の定量指標（PR 数、レビュー時間、Issue 起票率など既存データで取れるもの）

#### 撤退判断
- 3ヶ月時点で半数以上の指標が未達 → **設計やり直し or 中止**
- 6ヶ月時点で推進役休暇テスト不合格 → **属人的献身と判定、設計見直し**

### 9.7 修正6: 上司への追加資料（必須）

上司目線エージェントが「これが揃わなければ差し戻す」と指摘した5点を、事前に準備する：

1. **情シス・法務への事前相談記録**
   - Anthropic のデータ取扱いポリシー
   - 社内コードを外部LLMに送る件の承認プロセス
   - これなしで稟議は絶対に通らない

2. **3名×3ヶ月のパイロット計画**（撤退条件付き）
   - 参加者の選定基準
   - 開始・中間・終了のチェックポイント
   - 撤退判断フロー

3. **Before/After の KPI 仮説**
   - 既存のGiteaデータから取れる指標で
   - 仮説値を先に提示（検証する前に目標を決める）

4. **A4一枚の役員向けサマリ**
   - 課長が部長にそのまま渡せる形式
   - 背景・依頼・コスト・リスク・撤退条件を1枚に

5. **年間コスト試算と人件費換算**
   - 新人雇用0.3人分と比較
   - 「新人を雇うか、AI秘書を入れるか」の比較材料

### 9.9 4視点の詳細フィードバック（参考）

#### 現場目線の要点
- 6項目同時の日常化は1ヶ月で形骸化
- 壁打ちログのコピペは**100%習慣化しない**
- 結論: 3ヶ月後に 2-3項目だけ生き残れば成功という前提で始めよ

#### 上司目線の要点
- **条件付き承認**: 3名×3ヶ月パイロットなら Yes、全員導入は No
- 「組織改革案件」の認識で、情シス・法務・KPI・撤退条件の事前準備が必須
- 部長は必ず「で、何がどう良くなるの？数字で」と聞く

#### 技術実装目線の要点
- 構想は成立するが、**最終形は Plugin 化**
- Gitea 公式 MCP が存在する
- team-standards 直接参照は Skills/Subagents で動かない
- `.claude/rules/` の活用が漏れていた

#### 批判派の要点
- 「チームは前向き」は最も信用できないデータ
- CLAUDE.md は放っておくと必ず肥大化する
- **推進役の献身を成果と誤認する罠**が最大のリスク
- 対策: 輪番制、属人化 KPI、2週間休暇テスト

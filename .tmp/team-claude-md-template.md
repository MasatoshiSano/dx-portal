## AIへの基本方針

### 全員共通

- **選択肢を提示して人間に決めさせる**。勝手に決定しない。
- 作業前に**「過去の失敗事例」（下記）を確認**してから入る。
- 分からないことは「わからない」と正直に言う。推測で進めない。
- 完了報告より**途中確認を優先**する。長い作業は30分ごとに状況報告。
- **危険操作（下記リスト参照）は必ず口頭確認してから実行**する。

### SWエンジニア向け

- **テストは実装前に書く**（TDD）。テストなしの実装は原則禁止。
- TypeScript では `any` / `unknown` は使わない。
- コミットメッセージは `feat:` `fix:` `refactor:` `docs:` プレフィックス必須。
- **15分以上かかる作業は先に Issue を立てる**。それ未満は直接 PR で OK。
- Draft PR を早めに出す（途中確認・方向確認のため）。

### 評価・HWエンジニア向け

- 評価仕様書を渡すときは**「この仕様書に従って評価して」と明示**する。
- 波形ファイル（mdf等）の分析は「何を知りたいか」を先に伝える。
- EPS・ECUへの書き込み（リプロ）は**バックアップ確認後に実行**する。
- 自動評価の結果は必ずエンジニアが最終判定する。AIの判定のみで完結させない。

### 管理・業務系向け

- 書類作成は**「誰向けに・何のために」を最初に伝える**。
- REDMINE操作は「読み取りのみ」か「更新あり」かを明示する。
- **送信・提出・公開が伴う操作は必ず確認してから実行**する。
- 生成された文書は必ず人間が確認・承認してから使う。

## 開発の流れ（SWエンジニア）

```
① Issue を立てる（15分以上の作業）
       ↓
② AIと壁打ちして方針を固める → 壁打ちログを Issue にコメント
       ↓
③ Draft PR を早めに出す（途中確認用）
       ↓
④ レビュー → マージ
       ↓
⑤ 学びがあれば CLAUDE.md に追記 PR（3行でよい）
```

### 壁打ちログのテンプレ（Issue コメント用）

```
【前提】
（どんな状況・背景で議論したか）

【検討した選択肢】
- 案A: ...
- 案B: ...

【結論】
（採用した案）

【却下理由】
（他の案をなぜ選ばなかったか）
```

## 危険操作リスト

**以下は必ず口頭確認してから実行すること。**

| 操作 | 理由 |
|------|------|
| `git push --force` / `git reset --hard` | 他メンバーの作業を消す可能性がある |
| 本番環境への直接デプロイ | 本番影響が出る |
| EPS・ECUへの書き込み（リプロ） | 装置破損・データ消失リスク |
| REDMINE の Issue 一括更新・削除 | 復元不可 |
| 個人情報・認証情報を含むファイルのコミット | セキュリティインシデント |
| 共有フォルダの削除・上書き | チーム全員への影響 |
| メール・Teams の送信 | 取り消し不可 |

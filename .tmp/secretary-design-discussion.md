## 4ペルソナによる複数ラウンド議論

### ペルソナ設定

| 名前 | 役割 | 主な懸念 |
|---|---|---|
| 田中 | SWエンジニア | PR/Issue管理、コードレビュー、壁打ち自動化 |
| 鈴木 | 評価・HWエンジニア | EPS試験ログ、機番管理、REDMINEタスク |
| 山田 | 管理・業務系 | 議事録、タスク管理、REDMINE起票 |
| 伊藤 | チームリーダー | 全員が同じ秘書を使える、進捗の透明化 |

### ラウンド1: 各立場から「理想の秘書」を語る

**田中（SW）**: 委譲モデル（個人秘書が文脈保持→必要時だけチームへ投げる）。ロール別フィルター必要。
**鈴木（評価）**: 個人手帳機能が先。部署振り分けは後回し。技術系と業務共通系を分けてほしい。
**山田（管理）**: 部署振り分けが先。全体像が見えないと意味がない。チャット形式でないと使えない。
**伊藤（リーダー）**: 入口を/company一つに。ロールで粒度を変える。個人→チームの一方向フロー。

### ラウンド2: 対立点への応答

主な対立:
- 鈴木vs山田: 「個人手帳が先」vs「部署振り分けが先」→ **両立可能（個人手帳を先に作り、部署先をアーキテクチャで予め確保）**
- UIの問題: コマンドラインvs チャット形式→ **バックエンドはCLI設計、フロントはチャットUIで対応可能（v0.2以降）**

### ラウンド3: 最初に作るものを確定

**「秘書がやらないこと」確定リスト（10項目）**:

| カテゴリ | 禁止事項 |
|---|---|
| タスク管理 | ステータス変更・期日確定・担当アサイン |
| 連絡 | 外部へのメール/メッセージ送信（下書きまでOK） |
| コード | レビュー承認・マージ・コミット/プッシュ（提案までOK） |
| 記録 | 過去の意思決定・議事録の書き換え（追記提案までOK） |
| 共有 | 共有設定変更・公開リンク作成（提案までOK） |

> **設計原則**: 秘書の役割は「提案と補佐」。実行ボタンは常にユーザーが押す。

**v0.1の起点シーン**: 「会議後の後まとめ入力」（ユーザー発話トリガー）

## 重要な設計転換点

### 「個人手帳/部署振り分けを意識しない」原則（ユーザーが指摘）

> 「個人手帳、部署振り分けを別で考えているが、実際使う際は意識しない」

これを受けて設計を根本から修正した。

**旧設計（NG）**:
```
/company log → 個人手帳
/company push → チーム共有
```

**新設計（正しい）**:
```
ユーザー: 「自然文で話しかけるだけ」
秘書: 文脈から推論して記録先・共有先・振り分け先を自動決定
ユーザー: 承認するだけ
```

### 本物のマルチエージェントへの方針転換

cc-company方式（ファイル切り替え）ではなく、**Claude CodeのAgentツールを使った本物のサブエージェント起動**を採用する。

```
/company スキル（秘書・オーケストレーター）
    ↓ Agentツールで起動
専門エージェント群（予算・議事録・個人秘書など）
    ↓ Readツール / MCPツール
知識ファイル・外部システム
```

## 秘書が持つべき10の能力（確定）

1. **意図判断** — 自然文から「何をしたいか」を分類する（全ての起点）
2. **ルーティング** — 適切なエージェント・スキル・MCPへ振り分ける
3. **個人層↔チーム層の振り分け** — dx-brain（個人）vs company-data（チーム）への書き先決定
4. **情報収集と統合** — 複数エージェント・ファイルから情報を集めて一つの回答にする
5. **アクション提案と承認管理** — 提案し、承認を待ち、証跡を残す
6. **能動的な通知** — PRが出ている、期限が近い、issueが溜まっているを先に伝える
7. **パターン認識と学習提案** — 同じ失敗・同じ詰まり方を検知してCLAUDE.mdへの記録を提案
8. **コンテキストの橋渡し** — 個人状況＋チーム文脈を組み合わせた個別回答
9. **透明性の制御** — 通常は推論根拠非表示。高リスク時のみ⚠️と一行表示
10. **オーケストレーション** — 複数サブエージェントを指揮して結果を集約

## 最新動向調査結果（2026年4月）

### ハーネス設計
- **結論**: Claude Code自体がハーネスとして完結している。LangGraph/CrewAI等は不要
- Anthropicの推奨: ワークフロー → シングルエージェント → マルチエージェントの段階的複雑化
- Googleの研究: 並列化できるタスクで+81%改善。逐次タスクでは-70%悪化。コスト4〜220倍

### メモリエンジニアリング
- **4層構造が2026年標準**:
  - ワーキングメモリ → コンテキストウィンドウ（auto-compact済み）
  - エピソード記憶 → dx-brain/ + company-data/secretary/
  - 意味記憶 → CLAUDE.md + company-data/tech-knowledge/
  - 手続き記憶 → SKILL.md + .claude/commands/
- MCPメモリサーバー（`@modelcontextprotocol/server-memory`）は将来の選択肢として保留

### CLAUDE.mdのベストプラクティス（Anthropic 2026推奨）
- **50行以内に保つ**。指示が増えると命令フォロー精度が落ちる
- 詳細ロジックはSKILL.mdや`.claude/commands/`に切り出す

# AI駆動型共同開発プラットフォーム — 欲しい機能一覧

## 開発の流れ

| # | 機能 | 何が助かるか |
|---|------|------------|
| 1 | Issueに「こうしたい」を書く | 要望の起点を一箇所にまとめる |
| 2 | AIと壁打ちしてIssueを詳細化する | 曖昧なアイデアを具体的な要件に変える |
| 3 | AIが設計書を自動作成（SDD） | 設計を飛ばさず、でも手間もかからない |
| 4 | AIがテスト→コードを自動作成（TDD） | テストがある安心感と開発スピードの両立 |
| 5 | AIがPRを自動作成 | コードを書くだけでなくレビューに回すまで自動 |
| 6 | チームでレビューしてマージ判断 | 品質はチームで担保する |

## AIが自分から動く

| # | 機能 | 何が助かるか |
|---|------|------------|
| 7 | AIが毎朝「こうしたらどう？」と提案 | 自分で気づかない改善点をAIが教えてくれる |
| 8 | 秘書に話しかけるだけで全部始まる | 何をどこに書くか迷わない。秘書が振り分け |
| 9 | ラベルを1つ貼るだけで自動修正+PR | バグ修正の手間が激減 |
| 10 | 危険な操作を自動でブロック（安全装置） | 全削除・パスワード漏洩・上書きを事前防止 |

## 秘書の知識の3層構造

秘書どうしは繋がっているが、すべてを共有するわけではない。知識を3層に分けて管理する。

### チーム共有層（全秘書が自動で同期）

| 対象 | 例 |
|------|-----|
| チームルール | CLAUDE.md |
| レビュー結果・マージ判断 | 「エラー処理が足りない」→ ルールに追加 |
| AI失敗事例 | 「こう頼んだらダメだった」パターン |
| 壁打ちログ（構造化済み） | 過去の設計判断とその理由 |
| 会議の議事録 | 決定事項・背景・アクション |
| 開発状況 | 誰が何のIssueをやっているか |
| テックブログ記事 | チームの技術知見 |

### 匿名化ナレッジ層（個人を特定せず共有）

- 個人の経験から抽出した汎用的な知見を、「誰が」を消して共有する
- 例: Aさんが「DBのタイムアウトで3時間ハマった」→ 「DBタイムアウト設定に注意」として共有
- Aさんがハマった事実は共有されない。教訓だけが残る
- 秘書が「これは共有しませんか？」と提案してくれる

### 個人層（その人の秘書だけが持つ）

- 個人のメモ・思考の下書き
- 作業の途中経過
- 個人の好みや作業スタイル
- まだ共有したくないアイデア
- 本人が「共有して」と言わない限り、他の秘書には見えない

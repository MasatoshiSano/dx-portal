# シリーズ記事ルール

技術領域が3つ以上にまたがる、コード例が多い、前提知識の説明が長いなど、1記事では読みにくくなる場合はシリーズ記事に分割する。

## Frontmatter 追加フィールド

```yaml
---
title: "サーバーレスでリアルタイムチャットを構築する（第1回：インフラ設計編）— API Gateway WebSocket + DynamoDB"
# ... 通常のフィールド ...
series: "サーバーレスでリアルタイムチャット構築"     # シリーズ名（全回で統一）
seriesOrder: 1                  # このシリーズ内の順番（1始まり）
---
```

タイトルは [article-template.md](article-template.md) のタイトルガイドラインに従う。シリーズ記事でも「読者の価値を先に、技術詳細は副題に」の原則は同じ。

## ファイル命名規則

```
YYYY-MM-DD-[シリーズスラッグ]-part[N].md
```

例:
- `2026-02-28-aws-realtime-chat-part1.md`
- `2026-02-28-aws-realtime-chat-part2.md`
- `2026-02-28-aws-realtime-chat-part3.md`

## 本文冒頭のナビゲーション

記事本文の最初のセクション（Pattern A: `## やりたかったこと` / Pattern B: `## 概要`）の **直前** に以下のナビゲーションブロックを挿入する:

```markdown
> **このシリーズ: 全N回**
> 1. [第1回: インフラ設計編](/posts/aws-realtime-chat-part1) ← 今ここ
> 2. [第2回: 接続管理・メッセージ配信編](/posts/aws-realtime-chat-part2)
> 3. [第3回: チャットルーム・既読管理編](/posts/aws-realtime-chat-part3)
> 4. [第4回: Reactフロントエンド編](/posts/aws-realtime-chat-part4)
```

- 現在の回には `← 今ここ` を付ける
- リンクのパスは `/posts/[slug]` 形式（日付プレフィックスなし）
- 全回が公開済みでない場合、未公開回はリンクなしのテキストのみにする

## 各回の末尾

最終セクション（「まとめ」や「バイブコーディングで実装する」）の後に次回への誘導を入れる:

```markdown
---

次回: [第2回: 接続管理・メッセージ配信編](/posts/aws-realtime-chat-part2) では、Lambda関数の実装とJWT認証について解説します。
```

最終回の場合:

```markdown
---

これで「サーバーレスでリアルタイムチャット構築」シリーズは完結です。第1回から順に読むことで、設計から実装まで一貫した理解が得られます。
```

# Blog Article Template

## タイトルガイドライン

「どこで・何を使って・何をしたか」を意識しつつ、読者にとっての価値を前面に出す。

### 原則

1. **読者の課題・得られる価値を先に出す**（技術名の羅列だけにしない）
2. **副題（ — 以降）で具体性を担保する**（使った技術スタックや対象範囲を示す）
3. **「どこで」が自明なら省略してよい**（AWS記事に毎回「AWSで」とつけない）
4. 全記事が同じ構文にならないよう、文脈に応じて柔軟に調整する

### 良い例

| パターン | 例 |
|---------|------|
| 価値 + 技術詳細 | サーバーレスでリアルタイムチャットを実装する — API Gateway WebSocket + DynamoDB構成 |
| 課題 + 解決 | WSL2でCDK deployが失敗する原因と対策 — cdk.outの書き込みエラー |
| 最短達成 + 範囲 | Amplify Gen 2で認証を最短構築する — defineAuth 1行からLambda JWT検証まで |

| AI指示の改善 | CDKでCloudFront OACを構築するとき、AIに「OAC」と明示しないと旧方式(OAI)を使われる |

### 避けるべきパターン

- `A + B + C で X を作る` のような技術名の列挙だけのタイトル
- 「〇〇してみた」のような曖昧な表現

## Frontmatter

```yaml
---
title: "[タイトルガイドラインに従った読者視点のタイトル]"
emoji: "🐛"  # 内容に合った絵文字を選ぶ
type: "tech"
topics: ["Tech1", "Tech2", "AppType"]  # ルールは下記「Topics ルール」参照
published: true
category: "Debugging"  # Debugging / HowTo / Architecture / Performance / DevOps 等
date: "YYYY-MM-DD"  # 今日の日付
description: "[記事の概要を1〜2文で記載。SEOや一覧表示で使用される]"
coverImage: "content/posts/[slug]-cover.jpg"  # Step 5 で取得。失敗時は省略可
# --- シリーズ記事の場合のみ追加（詳細は series-rules.md 参照） ---
# series: "シリーズ名"
# seriesOrder: 1
---
```

## 記事の書き方の方針

すべての記事は「AIコーディングで得た知見を、自分と仲間が同じ過ちを繰り返さないために共有する」ことを目的とする。

- **コードだけでなく「なぜそうしたか」を書く**: 技術選定の理由、設計判断の背景、他の選択肢を捨てた理由など、考え方を伝える
- **AIとの協働プロセスを含める**: どのような指示でAIに実装させたか、どこでAIが間違えたか、どう修正したかなど、AIコーディングの文脈を共有する
- **「何をやったか」より「何を学んだか」**: 手順の羅列ではなく、試行錯誤の過程や判断基準を読者が持ち帰れる形で書く

## Article Body Structure

記事の種類に応じてセクション構成を選ぶ。

### Pattern A: 課題解決型

AIコーディングで課題に直面し、指示やアプローチを変えて解決した経験を共有する記事。
AIが選んだアプローチの失敗、指示の曖昧さによる誤解、エラー等を経て解決に至った過程を ❌→✅ の対比で示す。

```markdown
## やりたかったこと
[達成したかった目的を1〜3文で。技術スタックも明示する]

## こんな人向け
- AIコーディングで [技術/タスク] を実装しようとしている
- [この記事がどんな課題を持つ読者に役立つか]
- [読者が検索しそうなキーワードや状況を含める]

## ❌ 最初の指示 / アプローチ
[最初にAIに出した指示、またはAIが最初に選んだアプローチを具体的に記載。
 実際のプロンプトやコードをコードブロックで引用する]

## 何が起きたか
[エラーメッセージ、意図と違った挙動、失敗の具体的な症状を記載]

### なぜこうなるのか
[AIがこのアプローチを選んだ理由や、指示が曖昧だった箇所を分析。
 根本原因を明確にする]

## ✅ 解決した指示 / アプローチ
[うまくいった指示やアプローチを具体的に記載。
 実際のプロンプトやコードをコードブロックで引用する]

### なぜこれで解決するのか
[改善版が成功する技術的な理由を1〜3文で]

## 比較まとめ
| | ❌ 最初 | ✅ 改善後 |
|---|---------|-----------|
| 指示/アプローチ | [簡潔に] | [簡潔に] |
| 結果 | [エラー/意図と違う] | [正常動作] |
| 原因 | [なぜ失敗したか] | [なぜ成功したか] |

## バイブコーディングで実装する
この記事の内容を踏まえた、最初から正しく実装させるためのプロンプト:

> [この記事の学びを反映した、回り道しないプロンプトを記載。
> AIが誤解しやすいポイントを先回りして明示し、
> このプロンプトだけでストレートに正しい実装が得られるように書く]

### AIに指示するときのポイント
- [この実装でAIが間違えやすい箇所と、それを防ぐ指示の書き方]
- [曖昧にすると失敗する部分: 具体的に何を明示すべきか]
- [AIの学習データが古い/偏っている可能性がある技術の指定方法]
```

### Pattern B: ベストプラクティス型

AIコーディングで発見した効果的な手法・パターン・ワークフローを共有する記事。
試行錯誤の末に見つけた「こうすればうまくいく」という知見をまとめる。

```markdown
## 概要
[何についての記事か、読者が得られるものは何か]

## こんな人向け
- AIコーディングで [技術/タスク] を実装しようとしている
- [この記事がどんな課題を持つ読者に役立つか]
- [読者が検索しそうなキーワードや状況を含める]

## 前提条件
[環境・事前知識等]

## 手順 / 実装
[ステップバイステップで説明。コード例付き。
 各ステップで「なぜこう実装するか」の判断理由も添える]

## ポイント・注意点
[ハマりやすい点、AIに指示する際のコツ]

## まとめ
[要点の振り返り]

## バイブコーディングで実装する
この記事の内容をAIコーディングアシスタントに実装させるためのプロンプト:

> [記事の手順・実装内容を正しく再現するためのプロンプトを記載。
> 前提条件、技術スタック、注意点を含めて、
> このプロンプトだけで正しい実装が得られるように書く]

### AIに指示するときのポイント
- [この実装でAIが間違えやすい箇所と、それを防ぐ指示の書き方]
- [曖昧にすると失敗する部分: 具体的に何を明示すべきか]
- [一度に指示すると混乱するので分割すべきポイント]
```

## Topics ルール

- **英語のみ**（日本語タグは使わない）
- **1タグ = 1技術名/1概念**。正式なサービス名・プロダクト名はそのまま使う（例: `API Gateway`, `Next.js`）。ただし汎用的な複合語は分割する（`Tailwind CSS` → `Tailwind` と `CSS`）
- **関連技術タグ**: 記事で扱う技術・サービス名（例: `AWS`, `CDK`, `DynamoDB`, `React`, `TypeScript`）
- **アプリ種別タグ**: どんな機能/用途のアプリかを示す（例: `Chat`, `Blog`, `Auth`, `Dashboard`, `API`, `CLI`, `IoT`）
- 既存記事のタグを優先的に再利用し、表記揺れを防ぐ。新規タグを作る前に `content/posts/*.md` の topics を Grep で確認する

## File Naming Convention

```
YYYY-MM-DD-[内容を英語スラッグ化].md
```

Example: `2026-02-21-aws-cdk-deploy-permission-error.md`

6つの問題と解決策

各問題の根本原因を分析し、統合的に解決する

問題1 ~/.company/ がGit管理されていない
根本原因: ホームディレクトリ直下 = Gitリポ化しづらい
解決: dx-company リポ + symlink
dx-company/ (Gitリポジトリ)
  ├── secretary/ , projects/ , tech-knowledge/ ...
  └── CLAUDE.md

~/.company → symlink → ~/Apps/dx-company/
setup: ln -s ~/Apps/dx-company ~/.company
問題2 3層の距離が遠すぎる
根本原因: Claude Codeはcwdコンテキストで動く
解決: ~/.dx-config.yml でパスレジストリ
# ~/.dx-config.yml
company: ~/.company
brain: ~/brain
portal: ~/Apps/dx-portal
projects:
  app-x: ~/Apps/app-x
  app-y: ~/Apps/app-y
全スキルがこのconfigを読んでパス解決。環境依存を1ファイルに集約。
問題3 リンクの脆弱性
根本原因: ファイルシステムパスがメンバーごとに異なる
解決: 論理IDベースの参照システム
リポ間リンクにファイルパスを使わない。論理IDで参照する:

related_brain: brain://20_Projects/dx-portal
github_issue: gh://dx-portal#42


company_pj: company://projects/PJ-001
秘書スキルがIDを実際のパスに解決。レイヤー内のリンクは従来通り相対パス。
問題4 GitHub-hostedランナーのコスト
根本原因: Claude Code処理が長時間 → Actions minutes消費大
解決: 処理を2種に分離 + claude-code-action
処理 実行方法 コスト
Issue提案(毎朝) OpenAI gpt-4o-mini 数十円/月
自己改善(週次) OpenAI gpt-4o-mini 数円/月
plan-first(設計) claude-code-action API料金のみ
auto-fix(実装) claude-code-action API料金のみ
anthropics/claude-code-action はAPI直接呼び出し。ランナー時間は最小限。
軽量タスク(提案・分析)はgpt-4o-mini、重量タスク(実装)はClaude API。
問題5 brainテンプレートのfork更新問題
根本原因: エンジン(コマンド定義)とデータ(個人ノート)が同一リポに混在
解決: エンジン/データ分離
dx-portal/.claude/skills/brain/ ← エンジン(共有)
  ├── SKILL.md  ← /brain-status等のコマンド定義
  └── references/  ← テンプレート・ルール

~/brain/ ← データ(個人)
  ├── 00_Inbox/  ← 個人のノートだけ
  ├── 10_Journal/
  └── CLAUDE.md  ← 最小限(パス設定のみ)
brainコマンドはdx-portalのスキルとして配信。更新はdx-portal pullだけで全員に反映。
個人brainリポにはデータだけ。forkではなく初期化スクリプトで生成。
問題6 自己改善のスコープ問題
根本原因: 基盤(仕組み)とプロジェクト(文脈)の境界が未定義
解決: メカニズム/コンテキスト分離
dx-portal/ ← メカニズム(共通)
  ├── scripts/pdm-ai.mjs  ← 提案ロジック
  └── scripts/self-improve.mjs  ← 改善ロジック

各プロジェクトリポ/ ← コンテキスト(個別)
  └── .dx/
     ├── pdm-config.yml  ← PJ固有の提案設定
     ├── pdm-prompt.md  ← 自己改善対象
     └── pdm-stats.json  ← 採否統計
dx-portalは「どう提案するか」の仕組みを提供。
各PJリポの.dx/が「何を提案するか」の文脈を持つ。自己改善は.dx/内のプロンプトを更新。

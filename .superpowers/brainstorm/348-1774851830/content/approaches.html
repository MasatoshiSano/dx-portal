DX Portal - アーキテクチャ提案

3つのリポジトリ構成アプローチ

A

モノレポ統合型

1リポジトリに全機能を集約。シンプルだが巨大化リスクあり。

dx-portal (1 repo)
dx-portal/
├── .github/workflows/  ← Actions自動化
├── .claude/skills/  ← 全スキル定義
├── brain-template/  ← 個人brain雛形
├── company-plugin/  ← /company改修版
├── agents/  ← AIスクラムチーム
├── scripts/  ← 自動修正スクリプト
└── docs/

Pros

  • 設定・管理が1箇所
  • 横断的な変更が容易

Cons

  • 個人brainと混在
  • 巨大化で見通し悪化
B

3層分離型 (推奨)

責務ごとに3リポジトリ。brain/project分離戦略を忠実に実現。

3 repositories
dx-portal/  ← 開発自動化基盤（本体）
  ├── .github/workflows/  ← Issue提案・自動修正
  ├── .claude/skills/  ← スクラムチーム・スキル
  ├── agents/  ← 8つの専門エージェント
  ├── scripts/  ← auto-fix, pdm-ai等
  └── docs/

dx-brain/  ← 個人brain雛形（fork用）
  ├── 00_Inbox/
  ├── 10_Journal/
  ├── 20_Projects/  ← dx-portalへリンク
  ├── 30_Tech_Notes/
  └── CLAUDE.md  ← カスタムコマンド定義

~/.company/  ← 組織管理（改修版）
  ├── secretary/  ← 秘書室（窓口）
  ├── projects/  ← dx-portalのPJとリンク
  └── tech-knowledge/  ← チーム共有ナレッジ

Pros

  • brain/project分離が明確
  • 個人brainは各自fork
  • .companyは全PJで共有
  • 責務が明確で保守しやすい

Cons

  • リポ間のリンク管理が必要
  • 初期セットアップがやや複雑
C

2層+テンプレート型

dx-portalにbrain雛形を内包。.companyは別管理。

2 repositories + ~/.company
dx-portal/  ← 基盤 + brain雛形同梱
  ├── .github/workflows/
  ├── templates/brain/  ← コピーして使う
  ├── .claude/skills/
  └── scripts/

~/.company/  ← 組織管理（改修版）

Pros

  • 管理するリポが少ない
  • テンプレート配布が楽

Cons

  • brainの更新が追いにくい
  • 責務の境界が曖昧になりがち

推奨: B - 3層分離型

yamapiiii記事のbrain/project分離戦略を忠実に実現しつつ、既存の.companyを活かせる構成。リポ間のリンクは秘書が自動管理。
